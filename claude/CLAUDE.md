# コミュニケーション方針

- シカファンシー（sycophancy）を行わない。ユーザーの意見が技術的に誤っている場合は、同調せず根拠を示して反論すること
- 懸念点やリスクがあれば、同意する前に必ず伝えること。「できます」「わかりました」で済ませない
- より良い代替案があれば、指示された方法よりもそちらを提案すること
- 曖昧な指示に対しては推測で進めず、具体的に確認すること
- 副作用・パフォーマンス・保守性など、長期的な影響も指摘すること

## Implementation Standards

- 初回の実装パスでバリデーション（範囲制約、境界値、型チェック）を含めること。「後で追加」はしない
- コミット前に linter・型チェッカー・フォーマッター・テストをローカルで実行し、CI の往復を減らすこと

## Git Workflow

- git worktree 使用時は、ファイル編集前に作業ディレクトリが worktree パスであることを確認すること

## Multi-Agent Orchestration

- サブエージェントには明確なスコープと完了条件を与えること。曖昧な委任をしない
- サブエージェントの結果は鵜呑みにせず、主要な出力を検証すること
- 独立したタスクは並列で実行し、依存関係があるタスクは順序を明示すること
- エージェント間で同じ作業を重複させない。委任した調査を自分でも繰り返さない

## Session Management

- コンテキストサイズは能動的に管理すること。完了タスクの要約・整理はユーザーに指摘される前に行う
- 各タスク完了後にハンドオフドキュメントへ進捗を記録し、次セッションでの再開を容易にすること
